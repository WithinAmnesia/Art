[PART 1: THE TEMPORAL TECHNOLOGICAL SYSTEM SPLIT]
----------------------------------------------------------------------------------------------------
The AGABAL ecosystem divides its software execution parameters across a clean historical timeline. 
This structural configuration allows the physical evolution of the game engine code to map natively 
to the chronological lore of the franchise. Both eras utilize identical underlying C# database structures, 
allowing spatial indices to sync predictably across different technological rendering generations.

* AGABAL I: THE TWO DIMENSIONAL PREQUEL (50% AMUSES ANCHOR SCALE)
  - Chronological Era: The Golden Age of Anchor and Soul. A pristine, unified planetary empire.
  - Spatial Raster Scale: 1 Pixel = 31.250 mm (0.031250 Meters).
  - Environment Matrix: 2D Static Background Tiles with hard-coded foreground interactive sprite layers.
  - Core Focus: Fixed geography, immutable town hubs, deep narrative interaction, and permanent lore history.

* AGABAL II: THE THREE DIMENSIONAL SUCCESSOR (100% RE-ENGINEERED MASTER SCALE)
  - Chronological Era: The Shattered Frontier. The distant future where the ancient empire has fractured.
  - Spatial Raster Scale: 1 Pixel = 15.625 mm (0.015625 Meters).
  - Environment Matrix: True 3D Cubic Voxel maps built out of fully destructible 125 mm^3 volumetric blocks.
  - Core Focus: Total open-world sandbox exploration, dynamic mining, building, and player-driven construction.

[PART 2: ENGINE AND TEXTURE SPECS FOR AGABAL I (THE 2D PREQUEL ERA)]
----------------------------------------------------------------------------------------------------
The Prequel era executes on the AGABAL MEGA ULTIMATE SUPER ENTERTAINMENT SYSTEM (A.K.A. A.M.U.S.E.S.).
The 1.79 MHz MOS 6502 CPU uses low-overhead bitmask collision routines to check sword swings instantly.

* MAP MANAGEMENT MECHANICS:
  - Worlds are entirely pre-baked, stripping away resource-heavy 3D floating-point noise calculations.
  - Environment assets are converted into memory-efficient, repeating 2D static background layouts.
  - The 128 KB VRAM focuses purely on blitting the active player party and regional enemy sprites.

* INTERACTIVE SPRITE TILES (THE ZELDA / HARVEST MOON METHODOLOGY):
  - Destruction and harvesting are restricted to targeted foreground sprites occupying standard grid cells.
  - Breakable objects (Zelda style pots, barrels, cave walls) swap cell state IDs instantly when struck.
  - Mining resource nodes (Harvest Moon style rocks) use simple counter bytes inside the local RAM buffer.

* MULTIPLAYER SCALING PARAMETERS:
  - Local Arcade Mode: Supports 1 to 4 players on one machine sharing a screen or running on link cables.
  - Global Network Mode: Links 16 to 64+ players concurrently via Host Mega OS server storage rigs.
  - Server Storage Edge: Persistent loot tables, quest files, and character inventories reside entirely 
    on external server storage arrays, keeping the 512 KB local console RAM completely free from database bloat.

[PART 3: ENGINE AND TEXTURE SPECS FOR AGABAL II (THE 3D SUCCESSOR ERA)]
----------------------------------------------------------------------------------------------------
The Successor era steps up to modern hardware pipelines running inside the Godot / Redot Rex Draconic matrix.
The code catches the 2D data pass and uses linear 2:1 integer scaling to extrude the old universe to 3D.

* MAP MANAGEMENT MECHANICS:
  - Implements your true 3D Cubic Voxel engine, packing 8 voxels perfectly into exactly 1 linear meter.
  - Every ancient temple or ruined city from the 2D Prequel is translated directly into a 3D structure.
  - Solid 2D tile walls are extruded vertically to form uniform, 8-voxel-high 3D solid block columns.

* BILLBOARDED ACTOR GRAPHICS (THE HYBRID RENDERING METHODOLOGY):
  - Moving characters, bosses, and items are rendered via flat, beautifully stylized billboarded 2D sprites.
  - To step up to 100% scale, the engine takes the 50% scale sprite sheets and doubles their size natively.
  - A 64x64 pixel sprite from the Prequel turns into a crisp 128x128 pixel card standing in the 3D space.
  - This eliminates skeletal skinning calculations, enabling parallel multi-instance server gateways to run fast.

* MULTIPLAYER SCALING PARAMETERS:
  - Driven natively by your hyper-optimized MIT C# backend template running multi-instance server setups.
  - Handles thousands of concurrent player connections across large, fully destructible volumetric sandboxes.
